# UF/IF Debug Mapper - 한국어 가이드

## 스킬 개요

UF(단위 기능)와 IF(통합 기능) 문제를 코드 위치로 매핑하고, 체계적인 디버깅 계획을 생성하는 스킬입니다. 설계 단계 명세에서 코드 실행 지점으로의 추적성(Traceability)을 확보하고, 증상→원인→디버그 위치의 원인 분석 표를 제시합니다. 인간의 개입이 중심인 "Human-in-the-Loop" 디버깅을 지원합니다.

## 언제 사용하는가

다음과 같은 상황에서 사용합니다:
- **설계 후 구현 전:** UF/IF 사양을 바탕으로 디버그 맵 사전 생성
- **CI 실패:** 회귀 테스트 실패, 커버리지 미달 등 원인 파악 시
- **런타임 증상:** OOM, 수치 오차, 성능 저하, 스키마 오류 등 발생 시
- **주기적 유지보수:** 코드 변경 후 매핑 업데이트

## 입력

다음 정보를 제공해야 합니다:
- **UF 리스트:** `docs/uf/` (UF-01..UF-27 등)
- **IF 리스트:** `docs/if/` (IF-01..IF-08 등)
- **모듈 레이아웃:** 최상위 디렉토리 구조 (`src/`, `tests/` 등)
- **기본 정보:** 런타임 제약(Hz, VRAM 한도 등), 오류 로그/스택 트레이스 (선택)

## 출력물

다음 산출물이 생성됩니다:

| 산출물 | 설명 |
|--------|------|
| **UF→코드 매핑 테이블** | 각 UF별 핵심 모듈, 진입점(경로::심볼), 데이터 In/Out, 불변량 |
| **IF→코드 매핑 테이블** | 각 IF별 Producer/Consumer, 통신 방식, 실패 모드, 디버그 훅 |
| **증상→원인→위치 테이블** | 일반적 증상별 예상 근본원인, 최초 점검 위치, 재현 명령 |
| **디버그 플레이북** | 빠른 트리아지, 계측, 중단점, 프로파일링 절차 (Copy/Paste 가능) |
| **최소 수정 패턴** | 각 문제별 제안 Fix (diff 형식) |

## 슬래시 명령 예시

### 예시 1: 문서에서 디버그 맵 생성
```
docs/uf/ (UF-01..UF-20) 와 docs/if/ (IF-01..IF-08) 를 스캔하여
docs/uf_if_debug_map.md 를 생성하라.
각 UF/IF에 대해 src/ 하위 가장 가능성 높은 코드 진입점을 매핑하고,
불변량과 디버그 훅을 나열하라.
```

### 예시 2: 런타임 증상 트리아지
```
데이터 처리 파이프라인에서 간헐적으로 수치 오차가 발생한다.
docs/uf_if_debug_map.md 의 증상→원인 섹션을 업데이트하고,
중단점 위치와 추가해야 할 로그를 명시하라.
권고 사항은 최소·실행 가능 범위로 유지하라.
```

### 예시 3: IF 계약 실패 디버깅
```
IF-04 인터페이스에서 데이터가 수신되지만 파싱 값이 올바르지 않다.
디버그 맵의 IF 섹션에 직렬화/엔디언 체크와
디코드 함수 점검 목록을 추가하라.
최소 재현 체크리스트를 포함하라.
```

### 예시 4: 성능 목표 미달 트리아지
```
UF-12 (배치 처리)의 처리 시간이 목표(200ms)를 초과한다.
성능 프로파일링 계획을 수립하고
병목 가능성이 있는 진입점과 측정 항목을 매핑에 추가하라.
```

## 다른 스킬과의 연결

**함께 사용하는 스킬 흐름:**

1. **core-engineering** → **uf-if-debug-mapper**
   - Stage 7 UF 도출 후, 디버그 맵 사전 생성

2. **uf-chain-validator** → **uf-if-debug-mapper**
   - UF 체인 검증 후, 더 상세한 디버그 가이드 생성

3. **agent-orchestration** + **uf-if-debug-mapper**
   - Agent-1 (Architect)이 맵 생성
   - Agent-2 (Builder)가 코드 수정 지점 참고
   - Agent-3 (Verifier)가 증거 검증

## 4개 섹션 구조

### 섹션 1: UF→코드 매핑 (주요 테이블)

| UF-ID | Goal | 핵심 모듈 | 진입점(경로::심볼) | 데이터 In/Out | 불변량 | 테스트 | 증거 |
|------:|------|---------|-----------------|------------|---------|------|------|

**설명:**
- **핵심 모듈:** UF 관련 최소 파일 목록
- **진입점:** 실행 시작점 (함수/클래스명)
- **불변량:** 유지되어야 할 조건 (단위, shape, 단조 인덱스, 버퍼 크기)
- **증거:** 리포트/로그 경로

### 섹션 2: IF→코드 매핑 (인터페이스 테이블)

| IF-ID | 인터페이스 컨트랙트 | Producer | Consumer | 통신 방식 | 실패 모드 | 디버그 훅 |
|------:|-----------------|----------|----------|---------|---------|---------|

**공통 실패 모드:**
- 스키마 불일치 (필드 누락)
- 직렬화 정렬/엔디언 오류
- 타이밍/레이턴시 지터
- 공유 메모리 링 버퍼 지연
- 네트워크 멀티캐스트 조인/드롭

### 섹션 3: 증상→원인→디버그 위치

| 증상 | 예상 근본 원인 | 최초 점검 위치 | 빠른 확인 방법 | 재현 명령 |
|-----|-------------|------------|----------|---------|

**공통 증상:**
- OOM / VRAM 스파이크
- 수치 오차 또는 발산
- 처리 성능 저하 (목표 미달)
- 인터페이스 계약 불일치 (스키마 오류, null 값)
- 테스트 커버리지 게이트 실패
- 비결정론적 출력 (실행마다 결과 다름)

### 섹션 4: 디버그 플레이북

#### 4.1 빠른 트리아지 (5~10분)
1. 재현 단계 확인
2. 로그 및 환경 스냅샷 수집
3. 관련 UF/IF 식별
4. 매핑된 진입점으로 이동

#### 4.2 최소 계측 (Instrumentation)
- 구조화 로그 키: `uf_id, if_id, shape, dtype, units, frame, timestamp`
- 불변량 검증 assertion 추가 (디버그 플래그로 제어)
- 로그 저장: `reports/debug/<timestamp>/`

#### 4.3 중단점·감시점 (Breakpoints & Watchpoints)
- UF/IF별 권장 중단점:
  - `파일::함수` + 라인 힌트
- 감시 항목:
  - shape 변화 지점
  - 단위 변환 지점
  - 버퍼/링 인덱스 경계

#### 4.4 프로파일링 / 성능 분석
- **GPU:** `nvidia-smi` 스냅샷; Nsight Systems/Compute (필요 시)
- **CPU:** `cProfile`, `perf`, `py-spy`
- **메모리:** `memory_profiler`, `tracemalloc`
- **측정 항목:** peak 메모리, 처리량, 레이턴시/지터

## 증거 출력물 (필수)

```
reports/uf_if_debug_map.md          — 전체 디버그 맵 문서
reports/debug/<run>/logs.txt        — 실행 로그
evidence_pack/metrics.yaml          — pass/fail 임계값
coverage.xml                        — 테스트 변경 시
```

## Agent Orchestration 통합

| Agent | 역할 | 이 스킬의 사용처 |
|-------|------|-----------|
| Agent-1 (Architect) | 디버그 맵 생성/갱신 | 리포지토리 구조 스캔, UF/IF 명세 분석 |
| Agent-2 (Builder) | 최소 Fix 구현 | 매핑된 진입점에서 코드 수정 |
| Agent-3 (Verifier) | 증거 팩 검증 | 플레이북의 테스트/벤치마크 실행 |
