# Defense Proposal Prep 스킬

방산 연구 제안서 작성에 앞서 수행하는 **4단계 사전 분석 워크플로우**.
이 스킬의 최종 산출물인 "제안 전략 문서"는 이후 모든 제안서 작성 스킬(overview, rd-plan, tech-analysis 등)의
소스 자료로 직접 활용된다.

> 📖 **참조 파일**:
> - `references/rfp-analysis-template.md` — 요구사항 분류 및 평가 배점 분석 양식
> - `references/innovation-framework.md` — 혁신성·도전성 포지셔닝 프레임워크

---

## 전체 흐름

```
[입력] RFP 문서 (PDF/HWP/DOCX)
    ↓
Step 1. RFP 심층 분석
    - 명시 요구사항 추출
    - 평가 배점 분석
    - 암묵적 우선순위 파악
    ↓
Step 2. 기능 도출 + TRL 갭 분석  ←──┐
    - 요구사항 → 기능 매핑           │ 피드백 루프
    - 기확보/미확보 기술 분류         │ (기술 조사 결과로
    ↓                               │  기능 수정)
Step 3. 기술 벤치마킹 (병행)  ────────┘
    - 국내외 최신 기술 동향
    - 기존 기술의 한계점 도출
    - 제안 기술 후보 선정
    ↓
Step 4. 혁신성·도전성 포지셔닝
    - "기존 한계 → 제안 기술" 서사 구조 확정
    - 심사위원 관심 포인트 매핑
    ↓
[출력] 제안 전략 문서 (MD + PPTX 요약)
```

## Step 1 — RFP 심층 분석

### 1-1. 문서 로드

RFP 파일을 읽는다 (PDF의 경우 `pdf 스킬` 활용, HWP/DOCX는 텍스트 추출).
파일이 없으면 사용자에게 핵심 내용을 붙여넣도록 요청한다.

### 1-2. 명시적 요구사항 추출

`references/rfp-analysis-template.md`의 양식을 기반으로 다음을 추출한다:

**A. 성능 요구사항 (Performance Requirements)**
- 정량적 목표값 (KPI)을 모두 수치로 추출
- 단위, 측정 조건, 시험 방법이 명시된 경우 함께 기록

**B. 기술 범위 요구사항 (Scope Requirements)**
- 개발 대상 품목 / 시스템 목록
- 포함(In-Scope) / 제외(Out-of-Scope) 경계

**C. 일정/예산 제약 (Constraints)**
- 연구 기간, 단계 구분 기준
- 총 예산 한도 및 비목별 제약

**D. 납품물 요구사항 (Deliverables)**
- 시제품, 문서, 소프트웨어 등 납품 목록
- 규격 / 인증 요구사항

### 1-3. 평가 배점 분석 ← 핵심

RFP의 평가 기준표를 파악하여 아래 표를 작성한다:

| 평가 항목 | 배점 | 비율 | 평가 포인트 (심사위원이 보는 것) | 전략적 중요도 |
|---|---|---|---|---|
| 기술 혁신성 | XX점 | XX% | | 🔴 최고 |
| 기술 실현가능성 | XX점 | XX% | | 🟠 높음 |
| 추진 체계 | XX점 | XX% | | 🟡 중간 |
| 예산 적정성 | XX점 | XX% | | 🟡 중간 |

**전략적 중요도 기준:**
- 🔴 최고: 배점 상위 30% + 심사위원이 가장 집중하는 항목
- 🟠 높음: 차별화 가능성이 있는 항목
- 🟡 중간: 기본기 충족으로 충분한 항목

### 1-4. 암묵적 우선순위 파악

RFP의 배경 설명, 필요성 기술, 사업 목적 섹션을 분석하여:
- 발주처(방위사업청/각 군)가 **실제로 해결하고 싶은 운용상의 문제**가 무엇인지 파악
- "어떤 전장 환경에서, 어떤 적 위협에 대응하기 위한 것인가"
- 기술 외적 관심사 (국산화율, 중소기업 참여, 수출 가능성 등)

**판단 근거**: 이 단계에서 도출한 암묵적 우선순위가 이후 혁신성 포지셔닝의 핵심 서사가 된다.

## Step 2 — 기능 도출 + TRL 갭 분석

### 2-1. 요구사항 → 기능 매핑

각 성능 요구사항을 달성하기 위해 **필요한 기능(Function)**을 도출한다.

```
[성능 요구사항] → [기능] → [구현 기술 후보]
예시:
  "탐지거리 50km 이상" → 고감도 수신기, 신호처리 알고리즘, 안테나 설계
  → [기능 F1] 고감도 RF 수신, [기능 F2] CFAR 탐지 알고리즘, [기능 F3] 빔포밍
```

**기능 도출 표:**
| 요구사항 ID | 성능 목표 | 필요 기능 | 기능 ID | 비고 |
|---|---|---|---|---|

### 2-2. 기능 ↔ TRL 갭 분석

각 기능에 대해 현재 보유 기술 수준을 평가한다.
사용자에게 현재 보유 기술 수준을 확인하거나, 합리적 추정으로 초안 작성 후 확인 요청.

**갭 분석 표:**
| 기능 ID | 기능명 | 현재 TRL | 목표 TRL | 갭 | 기술 확보 방안 | 위험도 |
|---|---|---|---|---|---|---|
| F1 | 고감도 RF 수신 | 5 | 7 | 2 | 자체 개발 | 🟡 중 |
| F2 | CFAR 알고리즘 | 3 | 7 | 4 | 논문 → 구현 | 🔴 고 |

**기확보 기술** (TRL 갭 ≤ 1, 기존 과제 활용 가능): → 이후 `defense-proposal-tech-analysis` 스킬의 "기확보 기술" 섹션 소스
**미확보 기술** (TRL 갭 ≥ 2, 신규 개발 필요): → "미확보 기술" 섹션 소스

> ⚠️ 이 표는 Step 3 기술 조사 결과를 반영하여 **반드시 업데이트**해야 한다.

---

## Step 3 — 기술 벤치마킹 (Step 2와 병행)

Step 2의 기능 도출이 잠정 완료되면 즉시 병행 진행.
기술 조사 결과가 더 나은 접근을 제시하면 Step 2로 피드백하여 기능 정의를 수정한다.

### 3-1. 조사 범위 설정

Step 2에서 도출된 **미확보 기술** 및 **위험도 高** 기능을 우선 조사 대상으로 선정한다.

### 3-2. 국내외 기술 동향 조사

웹 검색으로 다음을 조사한다 (최신 3년 이내 자료 우선):

```
조사 항목 per 기술:
① 해외 선진 기술 현황
   - 주요 연구 프로그램/논문/특허
   - 달성된 성능 수치 (TRL, 벤치마크)
   - 핵심 접근 방법론

② 국내 현황
   - 관련 연구과제 (국방과학연구소, 대학, 기업)
   - 현재 기술 수준 및 한계

③ 기존 기술의 한계점  ← 혁신성 근거로 바로 연결
   - "현재 최고 기술이 해결하지 못하는 문제"를 명시
   - 한계의 원인 (물리적 제약, 계산량, 데이터 부족 등)
```

**기술별 조사 결과 카드:**
```
[기술명]
- 해외 최고 수준: [기관/논문] — [성능치], TRL X
- 국내 최고 수준: [기관] — [성능치], TRL X
- 핵심 한계: [한계 내용] — 원인: [원인]
- 유망 접근법: [방법론 1], [방법론 2]
- 소스: [출처 목록]
```

### 3-3. Step 2 피드백 루프

기술 조사에서 다음이 발견되면 Step 2 기능 도출을 수정한다:
- 더 효율적인 구현 방법이 있을 때 → 기능 F#의 구현 방안 변경
- 기존 기능 분해가 비효율적일 때 → 기능 병합/분리
- 완전히 새로운 접근법 발견 → 기능 추가

**피드백 발생 시 기록:**
```
변경: F2 CFAR 알고리즘 → F2-A CFAR + F2-B AI 기반 탐지 (분리)
이유: [논문명]에서 하이브리드 접근이 순수 CFAR 대비 탐지율 15% 향상 확인
```

## Step 4 — 혁신성·도전성 포지셔닝

`references/innovation-framework.md`를 읽고 프레임워크를 적용한다.

### 4-1. 혁신성 서사 구조 설계

혁신성은 "기존 기술의 한계 → 우리의 해결책"이라는 **인과 서사**로 표현해야 설득력이 있다.
(Step 3에서 수집한 기존 한계점이 이 서사의 재료가 된다.)

**혁신 서사 템플릿:**
```
① 현재 문제/한계: "[기존 기술]은 [조건]에서 [한계]의 문제가 있어,
                   [결과적 문제]가 발생한다."
② 기존 접근의 실패 이유: "이는 [근본 원인] 때문으로,
                          기존 [방법론들]로는 해결이 어렵다."
③ 우리의 제안: "본 연구는 [핵심 기술/방법론]을 통해
               [구체적 해결 방법]으로 이 문제를 해결한다."
④ 차별화 근거: "이는 [기존 방법] 대비 [정량적 개선치]의
               성능 향상을 목표로 하며, [특허/논문] 수준의
               독창성을 가진다."
```

### 4-2. 도전성 프레임 구성

심사위원이 "쉬운 연구"로 볼 수 있는 항목에 도전성 근거를 보강한다.

**도전성 매트릭스:**
| 기술적 난제 | 어려운 이유 (물리적/수학적 근거) | 극복 전략 | 기존 시도 실패 사례 |
|---|---|---|---|

### 4-3. 평가 배점 항목별 포지셔닝

Step 1에서 도출한 평가 배점 표와 연결:

| 평가 항목 | 우리의 강점 | 어필할 키워드 | 뒷받침 근거 |
|---|---|---|---|
| 기술 혁신성 | [강점] | [키워드 3개] | [기술 조사 결과] |
| 기술 실현가능성 | [강점] | [키워드 3개] | [기확보 기술, TRL] |

## 작업 규칙

- **병행 진행**: Step 2와 Step 3은 동시에 시작. Step 2 초안 → Step 3 조사 → Step 2 수정이 자연스럽게 순환.
- **혁신성 우선**: 기능 선택 기준에 "기존 기술이 못 하는 것"을 항상 포함. 혁신성은 마지막에 붙이는 게 아니라 설계 기준.
- **출처 명시**: 기술 조사 결과는 모두 출처(논문, 보고서, 발표자료) 포함. 날짜 기재 필수.
- **수치 일관성**: 이 문서에서 정의한 KPI, TRL 수치는 이후 모든 제안서 섹션에서 그대로 사용.
- **GLOBAL_RULES 준수**: Rule 1(승인 후 출력), Rule 2(MD 저장), Rule 3(판단 근거), Rule 4(파일 로그).
