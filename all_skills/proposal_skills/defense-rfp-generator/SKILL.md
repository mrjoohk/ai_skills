# Defense RFP Generator 스킬

RFP(공모 문서)가 없는 상황에서, **연구 주제명만으로 현실적인 합성 RFP를 생성**한다.

이 스킬이 생성한 RFP는 `defense-proposal-prep` 스킬의 입력으로 직접 사용할 수 있다.
실제 RFP가 나중에 공개되면 그 문서로 교체하면 된다.

> 📖 **참조 파일**:
> - `references/rfp-structure.md` — 방위사업청 형식 RFP 섹션 구조 + 예시
> - `references/search-strategy.md` — 분야별 검색 쿼리 전략 + 주요 데이터베이스

---

## 전체 워크플로우

```
[입력] 연구 주제 (과제명 + 선택: 분야, 기간, 예산)
    ↓
Step 1. 주제 해석 — 기술 도메인·군 운용 요구·개발 목표물 분류
    ↓
Step 2. 유사 프로그램 검색 — 국내외 5~7개 프로그램 웹 조사
    ↓
Step 3. 요구사항 합성 — 유사 프로그램에서 공통 요구사항 추출·정제
    ↓
Step 4. RFP 초안 작성 — references/rfp-structure.md 형식으로 작성
    ↓
Step 5. Docx + Md 생성 — 최종 파일 저장 및 검증
    ↓
[출력] YYMMDD_HHMM_synthetic_RFP_[주제약어].docx
        YYMMDD_HHMM_synthetic_RFP_[주제약어].md
```

## Step 1. 주제 해석

입력된 과제명에서 아래 4가지 분류를 수행한다.

### 1-A. 기술 도메인 분류
아래 표에서 해당하는 도메인을 1~3개 선택:

| 코드 | 도메인 | 관련 키워드 |
|------|--------|-------------|
| D-AI | AI·자율 | 강화학습, MARL, 자율 비행, 의사결정, 딥러닝 |
| D-UAV | 무인기 체계 | 드론, UAV, UAS, 무인기, 편대, 스웜 |
| D-RAD | 레이더·탐지 | SAR, AESA, EO/IR, 탐지, 추적, 식별 |
| D-NAV | 유도·항법 | GPS, 관성항법, 유도 조종, 정밀 유도 |
| D-COM | 통신·전자전 | 데이터링크, 재밍, LPD, 네트워크 |
| D-STR | 구조·추진 | 경량화, 복합소재, 엔진, 추진제 |
| D-SIM | 시뮬레이션·시험 | 학습 환경, HIL, SIL, 디지털 트윈 |
| D-C2 | 지휘통제 | 전술 데이터, 임무 계획, MUM-T |

### 1-B. 군 운용 요구 분류
- **공중 작전**: 공대공, 공대지, ISR, 전자전
- **지상 작전**: 정찰, 지뢰 탐지, 물자 보급, MOUT
- **해상 작전**: 대잠, 해상 감시, 기뢰 제거
- **우주·사이버**: 위성, 사이버 방어
- **공통 기반**: 시뮬레이션, 시험평가, 훈련체계

### 1-C. 개발 목표물 분류
- 소프트웨어 시스템 (알고리즘, 플랫폼, 프레임워크)
- 하드웨어 시스템 (센서, 기체, 탑재체)
- 하이브리드 (SW+HW 통합)

### 1-D. 연구 기간 및 TRL 설정
- 입력 없으면: 기간 3년, TRL 시작 2~3, 종료 TRL 6 기본값 사용
- 소프트웨어 과제: TRL 진행 빠름 (2→6이 3년 가능)
- 하드웨어 과제: 보수적으로 설정 (3→5 권장)

## Step 2. 유사 프로그램 검색

references/search-strategy.md를 읽어 도메인별 검색 쿼리를 확인한다.

### 검색 원칙
- **5~7개** 프로그램 수집 (너무 많으면 요구사항이 희석됨)
- 해외 3~5개 + 국내 1~2개 균형 유지
- 최신 프로그램 우선 (최근 5년 이내)
- KPI 수치가 명시된 자료 우선 확보

### 프로그램별 수집 정보
각 프로그램에서 아래를 추출한다:

| 항목 | 수집 이유 |
|------|-----------|
| 핵심 목표 (1~2문장) | 연구 목표 섹션에 활용 |
| 성능 요구사항 (PRD) | 기술 요구사항 섹션 작성 근거 |
| 달성 수치/KPI | 성능 목표 설정 근거 |
| 납품물/마일스톤 | 납품물 섹션 작성 |
| 연구 기간/예산 | 사업 규모 참조 |
| 평가 방법 | 평가 기준 섹션 참조 |

### 출처 신뢰도 등급 (높은 것 우선)
1. 공식 프로그램 페이지 (darpa.mil, afrl.af.mil, dapa.go.kr 등)
2. 공개 기술 보고서 (DTIC, NTIS, KISTI)
3. 학술 논문 (IEEE, AIAA, 한국항공우주학회)
4. 신뢰할 수 있는 방산 뉴스 (Breaking Defense, Aviation Week, 연합뉴스 등)

## Step 3. 요구사항 합성

수집한 유사 프로그램 데이터에서 **이 과제에 맞는 요구사항**을 합성한다.

### 요구사항 합성 원칙
- **공통 요구사항 우선**: 3개 이상 프로그램에서 반복 등장하는 요구사항을 핵심 요구사항으로 채택
- **현실적 목표치**: 최선 프로그램 수준의 80~90%를 목표치로 설정 (도전적이되 달성 가능)
- **국내 맥락 반영**: 해외 요구사항을 국내 방산 환경(방위사업청 체계, 국내 기반 기술)에 맞게 조정
- **중복 제거**: 유사한 요구사항은 통합하고 측정 가능한 형태로 정제

### 요구사항 분류 체계
```
성능 요구사항 (Performance Requirements, PR)
  PR-01, PR-02, ... : 정량적 수치로 표현 가능한 성능

기능 요구사항 (Functional Requirements, FR)
  FR-01, FR-02, ... : 시스템이 수행해야 하는 기능

인터페이스 요구사항 (Interface Requirements, IR)
  IR-01, IR-02, ... : 외부 시스템과의 연동 요구사항

시험·검증 요구사항 (Verification Requirements, VR)
  VR-01, VR-02, ... : 성능을 어떻게 검증할지
```

## Step 4. RFP 초안 작성

references/rfp-structure.md의 방위사업청 형식 7섹션을 채운다.

### 작성 시 중요 원칙

**명시적 요구사항 vs 암묵적 우선순위 구분**
- 명시: 숫자로 기술된 성능 임계값, 납품물 목록
- 암묵: 평가 배점에서 드러나는 우선순위 (배점이 높을수록 중요)
- 암묵적 우선순위를 RFP에 "힌트"로 포함시킬 것 (예: 평가 기준 배점 분배)

**현실적 배점 분배 (방위사업청 일반 기준)**
```
기술성 평가: 60~70점
  - 기술개발 계획의 구체성: 15~20점
  - 연구팀 역량: 10~15점
  - 혁신성·도전성: 10~15점
  - 기술적 접근방법: 15~20점

사업성 평가: 30~40점
  - 예산 계획의 적절성: 10~15점
  - 추진 일정의 현실성: 10점
  - 성과 활용 방안: 10~15점
```

이 배점을 합성 RFP의 평가 기준 섹션에 반영한다.

## Step 5. Docx + Md 생성

docx 스킬 방식(docx-js)으로 RFP 문서를 생성한다.

### 색상 테마
- PRIMARY: `#1B3A6B` — 섹션 헤더
- SECONDARY: `#2E6DA4` — 소제목
- ACCENT: `#C8962E` — 중요 요구사항 강조, 배점
- SUCCESS: `#2E7D51` — 필수 요구사항 표시
- WARNING: `#B85C1A` — 조건부 요구사항

### RFP 문서 스타일 포인트
- 공식 문서 톤: 명확한 동사형 요구사항 ("~를 지원해야 한다", "~를 제공해야 한다")
- 요구사항에 ID 부여 (PR-01 등)로 추적 가능하게
- 각 요구사항에 출처 각주 추가 (어느 유사 프로그램에서 유래했는지)

### 파일 저장
- 경로: `/mnt/proposal/YYMMDD_HHMM_synthetic_RFP_[주제약어].docx`
- 분석 요약: `/mnt/proposal/YYMMDD_HHMM_synthetic_RFP_[주제약어].md`
- 0.FilesUpdate.xlsx 로그 갱신 (GLOBAL_RULES Rule 4)

### 생성 후 검증
```bash
python /mnt/.skills/skills/docx/scripts/office/validate.py [파일경로]
```
