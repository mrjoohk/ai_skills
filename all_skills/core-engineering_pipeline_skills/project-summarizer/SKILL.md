# Project Summarizer

프로젝트 진행 이력을 핵심 중심으로 정리하여 `project_summary.md`를 생성하거나 업데이트한다.
과도한 분량보다 **핵심만 압축**하는 것이 목표다. 읽는 사람이 5분 안에 전체 흐름을 파악할 수 있어야 한다.

---

## 실행 순서

### Step 1 — 컨텍스트 수집

아래 소스들을 순서대로 확인하여 프로젝트 관련 정보를 수집한다.

1. **현재 대화 히스토리** — 지금까지 주고받은 내용에서 작업 흐름, 결정 사항, 오류, 해결책 추출
2. **workspace 파일 스캔** — 다음 파일들을 우선 확인:
   - `project_summary.md` (기존 요약본, 있으면 반드시 먼저 읽기)
   - `requirements.md`, `uf.md`, `if_list.md` (설계 산출물)
   - `GLOBAL_RULES.md` (프로젝트 규칙)
   - `0.FilesUpdate.xlsx`, `1.PromptsUpdate.xlsx` (작업 로그)
   - `review_docs/`, `output_docs/` 디렉토리 내 파일 목록
   - 최근 수정된 `.md`, `.py`, `.ts`, `.js` 파일 (최대 10개)
3. **사용자에게 확인** — 수집된 정보만으로 부족하면 아래 질문 중 필요한 것만 선택해서 묻는다:
   - 프로젝트 이름 / 목적
   - 현재 완료 단계
   - 특별히 강조하고 싶은 문제점이나 개선점이 있는지

### Step 2 — 기존 파일 확인

`project_summary.md`가 이미 있는 경우:
- 기존 내용을 먼저 읽고 구조를 파악한다
- **추가/변경된 내용만 병합**하여 업데이트한다 (기존 내용을 지우지 말 것)
- 업데이트 시 각 섹션 하단에 날짜 태그를 붙인다: `<!-- updated: YYYY-MM-DD -->`

`project_summary.md`가 없는 경우:
- `references/template.md`의 구조를 기반으로 새로 작성한다

### Step 3 — 내용 작성

`references/template.md`의 섹션 구조를 따른다. 작성 원칙:

- **사실 기반**: 추측이 아닌 실제 발생한 일만 기록
- **핵심 압축**: 섹션당 최대 5~7개 항목, 각 항목은 2~3줄 이내
- **판단 근거 포함**: 결정/선택 사항에는 왜 그 방법을 택했는지 1줄 추가
- **중복 제거**: 비슷한 내용은 통합하여 간결하게

### Step 4 — 파일 저장

- 저장 위치: 프로젝트 루트 (workspace 최상위)
- 파일명: `project_summary.md` (고정)
- 인코딩: UTF-8

### Step 5 — 완료 보고

저장 후 사용자에게 다음을 짧게 보고한다:
- 신규 생성인지 업데이트인지
- 어떤 섹션이 추가/변경되었는지
- 파일 링크

---

## 주의사항

- `project_summary.md`는 항상 **workspace 루트**에 저장한다 (review_docs나 output_docs 안에 넣지 말 것)
- 기존 파일이 있는데 내용이 부족하다고 판단되어도, 사용자 확인 없이 기존 섹션을 삭제하거나 재구성하지 말 것
- 정보가 불충분할 경우 그 섹션은 `(정보 없음 — 추후 보완 필요)` 로 표시하고 넘어간다
